# Window delivery is an at-least-once wake, not exactly-once cargo

Ambience is a continuing reader with chat-bound read and search tools over the durable Conversation Archive; the archive, not the Flue transcript, is its memory of the chat. A dispatched Window is therefore a wake-up call carrying convenient context, and delivering the same wake twice is harmless by construction: the agent re-reads, finds nothing new to do, and stays silent, as its participation policy already requires. This supersedes ADR 0006's admission-receipt state machine and manual resolution requirement, and amends ADR 0005's claim that every accepted message appears in a Window "admitted" to Ambience: every accepted live message still belongs to exactly one Window and is never evicted by count or age, but a Window's dispatch is at-least-once and may terminally fail.

A Window is `pending` until one dispatch attempt sequence settles it as `done` or, after bounded retries, `failed`. Failure is recorded and logged, never surfaced as a ceremony: the messages remain readable in the archive, the chat keeps flowing, and no state anywhere in the system is called Uncertain, blocks a chat, or awaits a human transport decision. Startup re-dispatches every `pending` Window with its original timestamps; whether a delayed reading still deserves a response is Ambience's judgment, not infrastructure's. Messages observed only through history backfill are archived and readable but never wake the agent. Verified GitHub webhook deliveries follow the identical contract with the provider delivery ID as their identity; the ingress ledger keeps transport deduplication of provider redeliveries and drops its Uncertain states.

The cross-database handoff ambiguity that motivated ADR 0006 is real — the application ledger and Flue's submission ledger cannot share a transaction, and Flue's public `dispatch()` mints a fresh random identity per call — but its worst consequence under this product is a duplicate wake, which the product absorbs. GitHub mutations are different: they act on the world, so the Operation Identity ledger, its reconciliation, and its explicit mutation doctor remain exactly as ADR 0004 and the existing audit language define them. ADR 0009's single application database stands; only its references to admission receipts and pre-admission Uncertain recovery are narrowed to the two-state Window ledger described here.
